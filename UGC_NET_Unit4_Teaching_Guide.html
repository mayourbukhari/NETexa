Processes & Threads

Process Concept · States · Control Block · Context Switching · Thread Fundamentals

🌍 What is a Process?

A process is an instance of a program in execution. It's the fundamental unit of work in an OS. Unlike a program (static code), a process is dynamic and has state, memory, and resources.

Process vs Program
  • Program: Static, passive — code on disk
  • Process: Dynamic, active — running instance with allocated resources
  • One program can have multiple processes (e.g., multiple browser windows)

📋 Process Control Block (PCB)

OS data structure containing all information about a process. Also called Task Control Block (TCB).

PCB Components
  • Process ID (PID): Unique identifier
  • Process State: New, Ready, Running, Waiting, Terminated
  • Program Counter: Next instruction address
  • CPU Registers: Current register values
  • Memory Management Info: Page tables, base, limit registers
  • I/O Status: Allocated devices, open files
  • Accounting Info: CPU time, memory used, priority

🔄 Process State Transitions

Processes move through distinct states as they execute. Understanding transitions is crucial for scheduling.

┌─────────┐ │ NEW │ └────┬────┘ │ Admitted ▼ ┌─────────────────┐ │ READY QUEUE │ ◄─┐ └────┬────────────┘ │ Time slice expired / I/O complete │ Scheduler │ │ dispatch │ ▼ │ ┌──────────────────┐ │ │ RUNNING ├──┘ └────┬───────┬────┘ │ │ I/O request │ Exit │ │ ▼ │ ┌───────────┐ │ │ WAITING │ │ │ (I/O Wait)│ │ └───────────┘ │ ▼ ┌──────────────┐ │ TERMINATED │ └──────────────┘
State Details
  • New: Process created but not yet admitted to ready queue
  • Ready: Waiting in queue to execute on CPU
  • Running: Currently executing on CPU
  • Waiting: Blocked for event (I/O, semaphore, etc.)
  • Terminated: Finished or killed, waiting for cleanup

🔀 Context Switching

OS saves current process's context (registers, program counter, memory info) into PCB, then restores another process's context. Expensive operation!

Context Switch Steps
1
Save current process context to PCB1
2
Update PCB1 state (e.g., Ready if suspended)
3
Select next process via scheduler
4
Restore PCB2 context (registers, PC, memory)
⚠️ Context Switch Overhead

Pure overhead — no productive work done. Time: microseconds to milliseconds (varies by architecture). Too many context switches = thrashing.

Threads vs Processes

🧵 Thread Fundamentals

A thread is lightweight process — shares process memory but has separate stack and registers.

Process:

  • Heavy resource usage
  • Separate memory space
  • Slower creation/switching
  • Isolated execution

Thread:

  • Light resource usage
  • Shared memory space
  • Faster creation/switching
  • Concurrent execution
// Java - Running threads within same process class MyThread extends Thread { public void run() { System.out.println("Thread: " + Thread.currentThread().getName()); } } public void main(String[] args) { for (int i = 0; i < 5; i++) { new MyThread().start(); // Concurrent threads } }

Multithreading Models

How many-to-many relationship between user and kernel threads is implemented.

One-to-OneUser thread → kernel threadTrue parallelism, one blocks not othersOverhead; thread limits

Memory Management

Allocation · Fragmentation · Paging · Segmentation · Virtual Memory · Page Replacement

📊 Memory Layout

How process memory is organized in typical OS with protection boundaries.

HIGH ADDRESS ┌─────────────────┐ │ KERNEL SPACE │ (Protected, for OS only) │ (OS, drivers) │ ├─────────────────┤ │ STACK │ (Local vars, return addresses) │ ↓ (grows down) │ │ │ │ ↑ (grows up) │ │ HEAP │ (Dynamic memory, malloc/new) ├─────────────────┤ │ DATA SEGMENT │ (Global/static vars) │ (initialized) │ ├─────────────────┤ │ TEXT SEGMENT │ (Read-only code) └─────────────────┘ LOW ADDRESS
💡 Stack vs Heap Collision

If stack grows too much (recursion) or heap grows too much (alloc), they collide → segmentation fault!

💔 Memory Fragmentation

Unwanted gaps in memory preventing efficient allocation.

External Fragmentation

Free space scattered in small holes. Process needing large contiguous block can't find it despite total free space available.

[ALLOC-20][FREE-10][ALLOC-30] [FREE-5][ALLOC-15][FREE-8] Process needing 15KB fails despite 23KB total free!

Internal Fragmentation

Allocated block larger than needed. Wasted space within block.

Allocate: [20 bytes] Given: [32-byte page] Wasted: [12 bytes]
Solutions
  • Compaction: Defragmentation of memory (expensive)
  • Paging: Fixed-size blocks eliminate external fragmentation
  • Segmentation: Variable-size but coherent units

📄 Paging - Fixed Size Chunks

Divide physical memory into fixed-size frames and logical memory into pages. Virtual address → Physical address via page table.

How Paging Works

Virtual Address: [Page Number | Offset]

Page number used to index page table → frame number. Final address: [Frame Number | Offset]

// Virtual address translation Virtual Addr = 1234 (decimal) Page Size = 4KB (4096 bytes) Page Number = 1234 / 4096 = 0 Offset = 1234 % 4096 = 1234 Page Table[0] = Frame 5 Physical Addr = 5 * 4096 + 1234 = 20480 + 1234 = 21714
Advantages
  • Eliminates external fragmentation
  • Simple allocation — any free frame works
  • Enables virtual memory > physical RAM
Disadvantage

Internal fragmentation (last page often not fully used). Page table overhead (memory + TLB lookup time).

🔍 Virtual Memory - Illusion of Unlimited Memory

OS manages memory such that process thinks it has entire address space available, though only fraction in RAM.

Virtual Memory Mechanism
  • Page Fault: Process accesses page not in RAM
  • Trap: Hardware generates interrupt
  • Fetch: OS loads page from disk to RAM
  • Resume: Process continues execution
⚡ Performance Note

Disk access ≈ 100,000x slower than RAM! Page fault = expensive. Minimize with good page replacement policy.

🗑️ Page Replacement Algorithms

When page fault occurs and RAM full, which page to evict? Goal: minimize page faults.

Process Synchronization

Critical Section · Mutex · Semaphore · Monitors · Race Conditions

⚠️ Race Condition - The Problem

Multiple processes access shared data simultaneously, outcome depends on execution order. Results unpredictable.

// Classic example: count++ (non-atomic) // Two processes increment shared variable Process 1: load count, R1 // R1 = 5 inc R1 // R1 = 6 store R1, count // count = 6 Process 2: load count, R2 // R2 = 5 (INTERLEAVED!) inc R2 store R2, count // count = 6 (WRONG! Should be 7) // If P2 interferes after P1's load, P2 overwrites P1's increment!

🛡️ Critical Section - Safe Access

Code segment accessing shared data. Only one process at a time allowed inside.

Requirements for Critical Section Solution
  • Mutual Exclusion: At most one in critical section
  • Progress: Process not blocked indefinitely
  • Bounded Waiting: Max iterations waiting before entry
do { ENTRY SECTION (acquire lock) CRITICAL SECTION (access shared data) EXIT SECTION (release lock) REMAINDER SECTION (other work) } while (true);

🔐 Mutex (Mutual Exclusion Lock)

Binary lock: locked or unlocked. Only one thread holds it. Simple but powerful.

// Pseudocode mutex lock; int count = 0; // Process 1 & 2 lock.acquire(); // Wait until unlocked count++; lock.release(); // Free lock for others
Real-world Mutex
  • Busy-waiting (Spin lock): Keep checking until acquired (wastes CPU)
  • Sleep-waiting: OS stores process, wakes when lock released (better)

📊 Semaphore - General Synchronization

Counter-based synchronization. Can represent multiple resources or complex conditions.

// Semaphore as resource counter semaphore S = N; // N resources available wait(S) { // Acquire resource while (S <= 0); // Spin if none available S--; } signal(S) { // Release resource S++; }

Binary Semaphore

Values 0 or 1 — essentially a mutex. Used for mutual exclusion.

Counting Semaphore

Values ≥ 0 — tracks N identical resources. Processes wait if depleted.

🎯 Producer-Consumer Problem (Classic)

Scenario: Producer generates data, consumer uses it. Buffer bounded size.
Solution Uses: Two semaphores (empty, full) + one mutex for buffer access.

Deadlock

Conditions · Prevention · Avoidance · Detection · Recovery

💀 What is Deadlock?

Situation where set of processes permanently blocked, each waiting for resource held by another. System makes no progress.

Real-world Example

Person A holds fork 1, waiting for fork 2.
Person B holds fork 2, waiting for fork 1.
Both starve forever! (Dining Philosophers problem)

📋 Four Necessary Conditions for Deadlock

ALL four must be true simultaneously. Break any one → no deadlock.

ConditionMeaningPrevention Method
Mutual ExclusionResource can't be sharedMake resources shareable (hard)
Hold & WaitProcess holds resource while waitingForce acquire all or nothing
No PreemptionCan't forcefully take resourceAllow preemption (risky)
Circular WaitCycle of waitersOrder resources → no cycles
💡 Deadlock = All 4 True

Remove even one condition → deadlock impossible!

🛡️ Deadlock Prevention vs Avoidance

Prevention

Guarantee deadlock impossible by design. Ensure at least one condition false always.

Method
  • Order resources strictly
  • Request all resources upfront
  • Preempt periodically

Avoidance

Allow all conditions, but deny requests leading to unsafe state.

Method (Banker's Algorithm)
  • Track resource availability
  • Check if request leads to safe state
  • Grant if safe, deny if unsafe

🔍 Deadlock Detection & Recovery

Allow deadlock to happen, then detect and recover.

Detection
  • Build resource allocation graph (RAG)
  • Cycle in graph → potential deadlock
  • Overhead: runs periodically (costly)
Recovery Methods
  • Abort deadlocked processes: Kill some → release resources
  • Resource preemption: Take resource, restart process
  • Rollback: Revert process to earlier checkpoint
⚠️ Trade-off

Prevention = safe but restrictive. Avoidance = complex overhead. Detection = reactive but recovers. Choose based on application!

File Systems

File Structure · Directory · Allocation Methods · Free Space Management · Reliability

📄 File Concept

Named collection of data/code stored on secondary storage. Provides OS abstraction for persistent storage.

File Attributes
  • Name: Human-readable identifier
  • Type: Determines how OS interprets (text, binary, executable)
  • Size: File length in bytes
  • Protection: Read/Write/Execute permissions
  • Timestamp: Creation/modification time
  • Location: Where data stored (block addresses)

💾 File Allocation Methods

How file blocks spread across disk. Different strategies balance access speed vs fragmentation.

MethodBlock ArrangementAccess SpeedFragmentation
ContiguousAll blocks adjacent on diskFast (random access)External—bad for growth
LinkedBlocks scattered; linked via pointersSlow (sequential only)No external fragmentation
IndexedIndex block contains pointers to all file blocksGood (random access via index)Index overhead
Modern Filesystems

Most use indexed allocation (inode/i-node). Fast random access, manageable overhead. Example: ext4, NTFS.

📋 Directory Structure

Hierarchical organization for managing files. Directory is file containing file metadata.

/ ├─ bin/ │ ├─ ls │ ├─ cat │ └─ mkdir ├─ home/ │ └─ user/ │ ├─ Documents/ │ ├─ Downloads/ │ └─ profile.txt ├─ etc/ └─ var/
Linear Directory vs Tree Directory
  • Linear: Flat list of files — simple but poor scalability
  • Tree (Hierarchical): Directories containing files/subdirs — organized, efficient search

🗑️ Free Space Management

OS tracks which disk blocks are available for new files/growth.

Bit Vector

Each bit = 1 block. Bit=1 means used, bit=0 means free. Simple, O(1) allocation.

Bitmap: 11010110100... Blocks: [U][U][F][U][F][U][U][U][F][U][F]...

Linked List

Free blocks linked together. O(1) to add free block, but search slow.

FREE → Block7 → Block15 → Block22 → ...

I/O Management

I/O Organization · Disk Scheduling · Buffering · Caching · Interrupt Handling

⚡ I/O Subsystem Architecture

OS mediates between CPU and I/O devices. Devices vary greatly; controllers provide standard interface.

CPU ◄────► Memory │ ▼ Memory Controller │ ▼ I/O Bus │ │ │ ▼ ▼ ▼ Disk NIC USB Ctrl Ctrl Ctrl │ │ │ ▼ ▼ ▼ Disk NIC USB Device
Device Controller

Hardware interface between device and bus. Manages device operations, handles interrupts.

📊 Disk Scheduling Algorithms

Minimize average access time by optimizing seek time and rotational latency. Arm scheduling selects order of servicing disk requests.

AlgorithmPolicySeek PatternPerformance
FCFSFirst come, first serveRandom jumpsUnfair; high seek time
SSTFShortest seekNearest nextBetter; starvation risk
SCANSweep arm like elevatorOne direction until end, reverseGood; fair
C-SCANCircular SCANOne direction, return to startVery good; uniform wait
LOOKSCAN but stop at last requestEfficient sweepExcellent; practical

🔄 Interrupt-Driven I/O

Device signals completion via interrupt. CPU pauses current task, services device.

Interrupt Timeline
1
I/O Request: CPU initiates disk read
2
CPU Continues: Doesn't wait; executes other tasks
3
Device Ready: Disk signals interrupt
4
Context Switch: CPU saves state, handles interrupt
5
Service: Transfer data; restore original context
vs Polling

Polling: CPU repeatedly checks device status → wasteful. Interrupts: Device notifies CPU → efficient!

💾 Caching & Buffering

Improve I/O performance by bridging CPU-device speed gap.

Cache

Store frequently accessed data. Transparent to apps. Write-through vs write-back policies.

Buffer

Temporary storage between fast and slow devices. Often queue of requests.

CPU Scheduling

Algorithms · Preemption · Priority · Real-time Scheduling · Performance Metrics

🎯 CPU Scheduling Goal

Select next process from ready queue to run on CPU. Maximize efficiency and fairness.

Key Metrics
  • CPU Utilization: % of time CPU busy (goal: high)
  • Throughput: Processes completed per time unit (goal: high)
  • Turnaround Time: Time from submission to completion (goal: low)
  • Wait Time: Time in ready queue (goal: low)
  • Response Time: Time from request to first response (goal: low)

⏱️ Scheduling Algorithms

AlgorithmPrincipleProsCons
FCFSFirst come, first servedSimple, fairLong processes block others; poor interactive response
SJFShortest job firstMin average wait timeFavorable to short jobs; starvation
Round Robin (RR)Time slice → ready queueFair; good response timeTime slice overhead; many context switches
PriorityHighest priority firstFavor important tasksStarvation; complexity
Multilevel QueueMultiple queues by priorityBalanced; flexibleComplex; overhead

🔄 Round Robin Detailed Example

Most common in modern OS. Each process gets time quantum (slice) then goes to back of queue.

// Example: Time quantum = 4ms // Processes: P1(8ms), P2(4ms), P3(2ms) READY QUEUE: [P1][P2][P3] Time 0-4: P1 runs → P1 has 4ms left READY QUEUE: [P2][P3][P1] Time 4-8: P2 runs → DONE READY QUEUE: [P3][P1] Time 8-10: P3 runs → DONE READY QUEUE: [P1] Time 10-14: P1 runs → DONE Total completion: 14ms Context switches: 3

⚡ Preemption vs Non-preemption

Preemptive

OS can interrupt running process. Used by Round Robin, Priority scheduling.

Advantage

Responsive; prevents CPU hogs

Non-preemptive

Process runs until completion/I/O. Used by FCFS, SJF (basic).

Advantage

Less overhead; simpler

📌 Modern Scheduling

Linux uses Completely Fair Scheduler (CFS). Windows uses priority-based preemptive with aging. Real-time systems (hard deadlines) use different strategies.

📝 Practice Questions

Question 1 of 12 0/12 (0%)
Question 1
What is the key difference between a process and a program?
Explanation: A program is passive code on storage. A process is a dynamic instance with state, memory, and resources. One program can spawn multiple processes.
Question 2
In what state can a process be interrupted by scheduler to run another process?
Explanation: Running process is interrupted (context switch) to give CPU to ready process. This is preemption. Waiting/New/Terminated don't occupy CPU, so no interruption needed.
Question 3
External fragmentation occurs in which memory allocation method?
Explanation: Contiguous allocation requires adjacent free blocks. Scattered free holes prevent allocation of large blocks → external fragmentation. Paging avoids this via fixed-size blocks.
Question 4
What happens when a page fault occurs and RAM is full?
Explanation: Page replacement is key to virtual memory. When RAM full, OS selects victim page (via algorithm like LRU), writes to disk, loads needed page. Process resumes.
Question 5
What is the solution to prevent race conditions in critical sections?
Explanation: Mutex/semaphore ensure only one process accesses critical section at a time. Disabling interrupts is costly; increasing CPU speed doesn't fix logic. Multithreading is necessary.
Question 6
For deadlock to occur, how many of the four conditions must be true?
Explanation: Deadlock requires ALL four conditions: Mutual Exclusion, Hold & Wait, No Preemption, Circular Wait. Break any one → deadlock prevented.
Question 7
Which disk scheduling algorithm minimizes average seek time but risks starvation?
Explanation: SSTF always selects nearest request → low average seek. But distant requests wait indefinitely (starvation). SCAN/C-SCAN address starvation via systematic sweeping.
Question 8
What metric measures time process spends in ready queue?
Explanation: Wait time = time in ready queue. Turnaround = total time submission to completion. Response = time to first output. CPU burst = actual execution time.
Question 9
Which memory management technique eliminates external fragmentation?
Explanation: Paging divides memory into fixed-size frames. Any free frame suitable — no fragmentation. Trade-off: internal fragmentation (last page may be partially unused).
Question 10
What is the primary advantage of context switching with threads vs processes?
Explanation: Threads share process memory space; no memory management updates needed during context switch. Processes have separate memory → more overhead. Threads faster to switch.
Question 11
In Round Robin scheduling with time quantum=5ms, what happens after 5ms?
Explanation: Round Robin: each process gets time slice (quantum). After slice expires, OS interrupts and schedules next ready process. Original process goes to back of queue if not done.
Question 12
What protection does the Page Table provide?
Explanation: Page table translates virtual→physical addresses & enforces access permissions. Each process has own page table → memory isolation. One process can't access another's memory.
