Database Fundamentals

What is a Database?

A database is an organized collection of structured data stored and accessed electronically. It provides tools for data storage, retrieval, modification, and protection while maintaining data integrity and consistency.

  • Persistence: Data survives program termination
  • Concurrency: Multiple users can access simultaneously
  • Query Language: SQL for data manipulation
  • Integrity: Constraints ensure data validity
  • Atomicity: All-or-nothing transaction semantics

Data Models

Data models define the structure of data and relationships between entities in a database.

Data Model Description Use Cases
Relational Tables with rows/columns, primary/foreign keys Most business applications, SQL databases
Hierarchical Tree structure with parent-child relationships File systems, XML documents
Network Nodes connected by labeled edges, many-to-many Complex relationships, social networks
Document (NoSQL) Semi-structured JSON/BSON documents MongoDB, flexible schemas, rapid development
Graph Nodes and edges for relationship queries Neo4j, recommendation engines, relationship analysis

Schema and Instances

Schema: The structural definition of the database (table definitions, columns, data types, constraints). It remains relatively static.

Instance: The actual data contained in the database at a particular point in time. It changes frequently.

-- Schema Definition CREATE TABLE Students ( StudentID INT PRIMARY KEY, Name VARCHAR(100) NOT NULL, Age INT, GPA DECIMAL(3,2) ); -- Instance (Sample Data) INSERT INTO Students VALUES (101, 'Alice', 20, 3.8), (102, 'Bob', 21, 3.5);

DBMS Architecture

External View (User Interface) ↓ Conceptual Schema (Logical DB Structure) ↓ Internal Schema (Physical Storage) ↓ Storage Layer (Disk/Memory)

Three-Level Architecture:

  • External Level: Multiple user views of data
  • Conceptual Level: Unified logical schema independent of storage
  • Internal Level: Physical storage organization on disk

Integrity Constraints

Rules that ensure data quality and consistency:

  • Entity Integrity: Primary key cannot be NULL and must be unique
  • Referential Integrity: Foreign keys must reference valid primary keys
  • Domain Integrity: Data types and formats are enforced
  • Key Constraints: Uniqueness constraints on designated columns
  • Check Constraints: Custom business logic validation

SQL & Query Processing

SQL Fundamentals

SQL (Structured Query Language) is the standard language for database operations. It has four main categories:

Category Commands Purpose
DML SELECT, INSERT, UPDATE, DELETE Manipulate data
DDL CREATE, ALTER, DROP, TRUNCATE Define database structure
DCL GRANT, REVOKE Control access permissions
TCL COMMIT, ROLLBACK, SAVEPOINT Transaction control

JOIN Operations

JOINs combine rows from multiple tables based on related columns:

-- INNER JOIN: Only matching rows SELECT S.Name, C.CourseTitle FROM Students S INNER JOIN Courses C ON S.StudentID = C.StudentID; -- LEFT JOIN: All left table rows + matches SELECT S.Name, COUNT(E.CourseID) FROM Students S LEFT JOIN Enrollments E ON S.StudentID = E.StudentID GROUP BY S.StudentID; -- FULL OUTER JOIN: All rows from both tables SELECT * FROM Table1 FULL OUTER JOIN Table2 ON Table1.ID = Table2.ID;

Subqueries & Aggregation

Nested queries and aggregate functions for complex data analysis:

-- Subquery in WHERE clause SELECT Name, Salary FROM Employees WHERE Salary > (SELECT AVG(Salary) FROM Employees); -- GROUP BY with HAVING SELECT Department, COUNT(*) as Count, AVG(Salary) FROM Employees GROUP BY Department HAVING COUNT(*) > 5; -- Correlated Subquery SELECT E.Name FROM Employees E WHERE E.Salary > (SELECT AVG(Salary) FROM Employees E2 WHERE E2.Department = E.Department);

Query Optimization

Strategies to improve query performance and reduce execution time:

  • Use Indexes: Create indexes on frequently searched columns
  • Avoid SELECT *: Select only needed columns
  • Minimize JOINs: Join only necessary tables
  • Use WHERE Early: Filter rows before joining
  • Avoid Functions on Indexed Columns: Prevents index usage
  • Use LIMIT: Reduce result set size when possible

Window Functions

Perform calculations across a set of rows, useful for ranking and running totals:

-- Ranking employees by salary within department SELECT Name, Salary, Department, RANK() OVER (PARTITION BY Department ORDER BY Salary DESC) AS Rank FROM Employees; -- Running total of sales SELECT Date, Amount, SUM(Amount) OVER (ORDER BY Date) AS RunningTotal FROM Sales;

Data Normalization

Why Normalization?

Normalization is the process of organizing attributes into tables to:

  • Eliminate data redundancy
  • Reduce storage space
  • Improve data integrity and consistency
  • Facilitate data updates without anomalies
  • Simplify database maintenance
  • Insertion Anomaly: Cannot insert data without certain attributes
  • Deletion Anomaly: Deleting one fact removes unrelated data
  • Update Anomaly: Updating one value requires multiple changes

Normal Forms Overview

Normal Form Condition Key Achievement
1NF Atomic attributes only (no multi-valued attributes) Eliminate repeating groups
2NF 1NF + No partial dependencies Remove partial FD on composite keys
3NF 2NF + No transitive dependencies Remove non-key attribute dependencies
BCNF 3NF + Every determinant is a candidate key Strong normalization, removes all anomalies
4NF BCNF + No multi-valued dependencies Handle complex relationships

First Normal Form (1NF)

Definition: All attributes must be atomic (indivisible). No repeating groups or multi-valued attributes.

Before 1NF:

StudentID | Name | Courses 1 | Alice | Math, Physics, Chemistry 2 | Bob | English, History

After 1NF:

StudentID | Name | Course 1 | Alice | Math 1 | Alice | Physics 1 | Alice | Chemistry 2 | Bob | English 2 | Bob | History

Second & Third Normal Forms

2NF: 1NF + No partial dependencies (all non-key attributes depend on entire primary key)

3NF: 2NF + No transitive dependencies (non-key attributes don't depend on other non-key attributes)

Student(StudentID, Name, DepartmentID, DepartmentName, DepartmentLocation)

Problem: DepartmentName and DepartmentLocation depend on DepartmentID (transitive)

Solution:

Student(StudentID, Name, DepartmentID)

Department(DepartmentID, DepartmentName, DepartmentLocation)

Functional Dependencies

Definition: Attribute Y is functionally dependent on attribute X if each value of X determines exactly one value of Y.

  • Partial Dependency: Y depends on part of composite key X
  • Transitive Dependency: X → Y and Y → Z, therefore X → Z
  • Full Dependency: Y depends on entire composite key

Indexing & Query Optimization

Indexing Concepts

An index is a data structure that improves the speed of data retrieval. It's a sorted collection of references to data organized for fast searching.

  • ✓ Speeds up SELECT queries and WHERE clauses
  • ✗ Slows down INSERT, UPDATE, DELETE (must maintain index)
  • ✗ Consumes additional storage space

B-Tree Indexing

B-Tree: Most common index structure. Self-balancing tree where each node can have multiple children.

[40] / \ [20, 30] [50, 60, 70] / | \ / | | \ 10 25 35 45 55 65 75

Properties:

  • Logarithmic search time O(log n)
  • Balanced tree maintains this guarantee
  • All leaf nodes at same level
  • Supports range queries efficiently
  • Each node typically matches disk block size

Hash Indexing

Hash Index: Uses hash function to map keys to bucket addresses. Fast for equality searches but poor for range queries.

Hash Function Example: h(key) = key mod bucket_count

Feature B-Tree Hash
Equality Search O(log n) O(1) average
Range Search O(log n + result) Not efficient
Insertion/Deletion O(log n) O(1) average
Space Utilization Better Can be wasteful

Query Optimization Techniques

Query Optimizer: Component that determines the most efficient way to execute a query.

  • Index Selection: Choose appropriate indexes for WHERE clause conditions
  • Join Order: Reorder joins to minimize intermediate results
  • Predicate Pushdown: Apply filters early before joins
  • Materialization: Cache intermediate results if reused
  • Parallel Execution: Execute independent operations concurrently

Compare multiple execution plans and choose lowest cost based on:

  • I/O operations
  • CPU time
  • Network transmission (distributed DB)

EXPLAIN and Query Plans

Examine query execution plans to understand performance:

EXPLAIN ANALYZE SELECT * FROM Orders WHERE OrderDate > '2024-01-01'; -- Output shows: -- Seq Scan on Orders (cost=0.00..156.00) -- Filter: (OrderDate > '2024-01-01') (rows estimated) -- With index: CREATE INDEX idx_order_date ON Orders(OrderDate); -- Now uses: Index Scan on idx_order_date (faster)

Transactions & Concurrency Control

ACID Properties

ACID are fundamental guarantees for database transactions:

Property Definition Example
Atomicity All-or-nothing: transaction completes fully or rolls back completely Money transfer: debit and credit both succeed or both fail
Consistency DB moves from one valid state to another; all constraints satisfied Account balance never goes negative if constrained
Isolation Concurrent transactions don't interfere with each other Two simultaneous transfers don't cause balance errors
Durability Committed data persists even after system failures Data survives crashes once COMMIT confirmed

Concurrency Problems

Race Conditions: When multiple transactions access same data concurrently:

Problem Description Scenario
Dirty Read Transaction T2 reads uncommitted changes from T1 T1 updates balance but rolls back; T2 used wrong value
Lost Update Concurrent updates overwrite each other T1 and T2 both read X=10, both add 5, final result 15 (lost one +5)
Phantom Read Transaction T2 inserts rows visible to T1 in second read T1 selects age>20, T2 inserts new age>20 record, T1 re-queries finds different result
Non-Repeatable Read Uncommitted update changes value between reads T1 reads X=10, T2 updates X=20, T1 re-reads gets 20

Locking Mechanisms

Lock: Mechanism to prevent concurrent access to shared resources.

  • Shared Lock (Read Lock): Multiple transactions can read; no writes allowed
  • Exclusive Lock (Write Lock): Only one transaction can access; blocks all other locks
  • Two-Phase Locking (2PL): Growing phase (acquire locks) then shrinking phase (release locks)
  • Deadlock: When T1 waits for resource held by T2 which waits for resource held by T1
| Shared | Exclusive Shared | ✓ | ✗ Exclusive | ✗ | ✗

Isolation Levels

SQL standard defines four isolation levels (in increasing strictness):

Level Dirty Read Non-Repeatable Phantom Read
Read Uncommitted ✗ Possible ✗ Possible ✗ Possible
Read Committed ✓ Prevented ✗ Possible ✗ Possible
Repeatable Read ✓ Prevented ✓ Prevented ✗ Possible
Serializable ✓ Prevented ✓ Prevented ✓ Prevented

Recovery & Backup Strategies

Why Database Recovery?

Systems can fail for various reasons: power outages, disk errors, software crashes. Recovery mechanisms ensure data isn't lost and database returns to consistent state.

  • Transaction Failure: Within transaction (constraint violation)
  • System Failure: Sudden shutdown (main memory lost, disk OK)
  • Disk Failure: Permanent storage corruption (catastrophic)

Write-Ahead Logging (WAL)

WAL: Before writing data to disk, write a log record. Ensures durability and enables recovery.

Transaction → Log Buffer → Disk Log ↓ Data Buffer (can be cached) ↓ Write to Data Disk

Recovery Process:

  • Redo: Reapply committed transactions that weren't written to disk
  • Undo: Rollback uncommitted transactions that were partially written

Checkpoints

Checkpoint: Periodic point where all committed data is written to disk and a checkpoint record is created.

Benefits:

  • Reduces recovery time by marking a known consistent state
  • No need to redo transactions older than most recent checkpoint
  • Simplifies crash recovery process
-- Timeline Example T1 Start T2 Start T1 Commit Checkpoint (writes T1 to disk) T2 Update (in memory only) CRASH! After recovery: T1 is restored, T2 is rolled back No need to redo T1 (it's already in checkpoint)

Backup Strategies

Strategies to protect against permanent data loss:

Backup Type Description Recovery Time
Full Backup Complete copy of entire database Longer backup, faster recovery
Incremental Only changes since last backup Faster backup, slower recovery
Differential Changes since last full backup Balance of backup/recovery speed
Continuous Real-time replication to secondary copy Near-zero data loss (RTO/RPO)
  • 3 copies of data
  • 2 on different storage media
  • 1 copy offsite/remote location

Disaster Recovery Planning

RTO (Recovery Time Objective): Maximum acceptable downtime

RPO (Recovery Point Objective): Maximum data loss acceptable (measured in time)

-- Example SLA -- Critical system: RTO = 1 hour, RPO = 15 minutes -- Means: Must be running within 1 hour and lose at most 15 min of data -- Strategies to achieve this: -- - Database replication to multiple sites -- - Transaction log shipping (redo logs to standby) -- - Continuous backup/mirroring -- - Failover automation

Advanced Database Topics

NoSQL Databases

NoSQL: Non-relational databases designed for scalability, flexibility, and high performance with unstructured/semi-structured data.

Type Example Use Case Trade-off
Document MongoDB Content management, mobile apps No ACID guarantees, complex queries
Key-Value Redis, Memcached Caching, sessions, real-time data Limited query capabilities
Graph Neo4j Social networks, recommendations Scaling challenges
Time-Series InfluxDB, TimescaleDB Metrics, monitoring, IoT Specialized for time data

Distributed Databases

Distributed DB: Database spread across multiple nodes/locations for availability and scalability.

  • Horizontal Partitioning (Sharding): Split data across nodes by key range
  • Vertical Partitioning: Different columns on different nodes
  • Replication: Copy data to multiple nodes for redundancy

Choose only 2 of 3: Consistency, Availability, Partition Tolerance

  • Traditional RDBMS: Choose CA (no partition tolerance)
  • Most distributed systems: Choose AP or CP

Cloud Databases

Managed cloud databases abstract infrastructure, provide scalability and high availability:

  • Azure SQL Database: Managed relational DB with auto-scaling
  • Amazon RDS: Relational database service (MySQL, PostgreSQL, Oracle, MariaDB)
  • Google Firestore: Managed NoSQL document database
  • Cosmos DB: Multi-model globally distributed database
  • No infrastructure management
  • Automatic backups and recovery
  • Global replication and failover
  • Pay-per-use pricing model

Data Warehousing & OLAP

OLTP vs OLAP:

Aspect OLTP OLAP
Purpose Online transactions (operational) Analytical queries (business intelligence)
Queries Simple, short-running Complex, long-running aggregations
Data Current state, normalized Historical trends, denormalized
Volume Frequent small updates Batch loads, read-intensive

Data Privacy & Security

Critical aspects of modern database management:

  • Encryption: Encrypt data at rest (storage) and in transit (network)
  • Authentication: User verification before access (passwords, multi-factor)
  • Authorization: Role-based access control (RBAC) and permissions
  • Audit Logging: Track who accessed what data and when
  • Data Masking: Hide sensitive data in non-production environments
  • Compliance: GDPR, HIPAA, PCI-DSS regulations
  • Parameterized queries prevent SQL injection
  • Input validation and sanitization
  • Least privilege principle for database users

Practice MCQs - Database Systems

1. Which normal form ensures that no attribute depends on part of a composite primary key?

Explanation: 2NF requires 1NF compliance plus removal of partial dependencies, meaning all non-key attributes must depend on the entire primary key, not just part of it.

2. What is the correct SQL syntax to create an index on the 'Email' column of the 'Users' table?

Explanation: The correct SQL syntax is CREATE INDEX index_name ON table_name(column_name). The index name is conventionally prefixed with 'idx_' for clarity.

3. Which ACID property ensures that committed data survives system failures?

Explanation: Durability ensures that once a transaction is committed, the data is permanently stored and won't be lost even if the system crashes immediately after.

4. What does a FULL OUTER JOIN return?

Explanation: FULL OUTER JOIN combines all rows from both tables. Rows without matches in either table show NULL values for columns from the table with no match.

5. Which isolation level prevents all dirty reads, non-repeatable reads, and phantom reads?

Explanation: Serializable is the strictest isolation level and prevents all concurrency anomalies (dirty reads, non-repeatable reads, and phantom reads) by essentially making transactions execute sequentially.

6. What is the primary advantage of B-Tree indexing over Hash indexing?

Explanation: B-Trees maintain sorted order allowing efficient range queries (e.g., WHERE age BETWEEN 18 AND 65), while Hash indexes are optimized only for exact match queries and cannot efficiently handle range conditions.

7. Which statement best describes functional dependency X → Y?

Explanation: Functional dependency X → Y means that if two rows have the same X value, they must have the same Y value. This is fundamental to normalization and data consistency.

8. What is the purpose of Write-Ahead Logging (WAL)?

Explanation: WAL is a crash recovery technique where transaction log entries are written to disk before the actual data changes. This ensures that even if the system crashes, committed transactions can be reconstructed from the log.

9. In Two-Phase Locking (2PL), what happens after the shrinking phase begins?

Explanation: 2PL has two phases: Growing (acquire locks) and Shrinking (release locks). Once a transaction starts releasing locks (shrinking phase), it cannot acquire any new locks. This prevents cascading aborts and ensures serializability.

10. Which of the following best describes denormalization?

Explanation: Denormalization is the deliberate introduction of redundancy (like storing duplicate data) to improve query performance by reducing joins needed. Common in data warehouses (OLAP systems) where analysis speed is prioritized over insertion speed.

11. What does the CAP Theorem state about distributed systems?

Explanation: The CAP Theorem proves that distributed databases can guarantee at most two of the three properties: Consistency (all nodes see same data), Availability (system always responds), and Partition Tolerance (system works despite network splits). Most systems choose to sacrifice strong consistency.

12. Which of the following is a characteristic of NoSQL databases like MongoDB?

Explanation: NoSQL databases like MongoDB trade some ACID guarantees for flexibility and scalability. They support flexible schemas where different documents can have different fields, making them ideal for applications where data structure evolves rapidly.
